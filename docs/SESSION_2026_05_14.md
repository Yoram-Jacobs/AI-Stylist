### Patch 6 ‚Äî Optimistic-first delete in closet UX

`Closet.jsx` `handleDelete` and `ItemDetail.jsx` `onDelete` were
inverted: they `await api.deleteItem(id)` before updating
`closetStore`. New flow:

1. Snapshot the item(s) being deleted.
2. `store.remove(id)` immediately (UI updates).
3. `nav('/closet')` / close confirm dialog.
4. Fire `api.deleteItem(id)` in the background.
5. On rejection, `store.upsert(snapshot)` and toast the error.

Result: the closet feels instant; the toast confirms (or reverses)
seconds later. Restores the pre-regression "supercharged" UX
described by the user as "tap and done".

### Patch 7 ‚Äî Derive `original_image_url` from `image_base64` on `/closet` create

Was: the AddItem flow sent `image_base64` to `POST /closet` but never
`original_image_url`. The save handler stored `original_image_url =
None`. Production never noticed because the old HF segmentation
branch (removed in Patch 4) was populating `segmented_image_url` as a
side effect. After Patch 4, legacy-path saves landed with no image
URL at all ‚Üí blank closet card.

Fix is in `app/api/v1/closet.py:create_item`: when `payload.image_base64`
is present and `payload.original_image_url` is empty, derive

    original_image_url = f"data:{image_mime};base64,{image_base64}"

before persisting. Verified live: new items render thumbnails
immediately.

### Patch 8 ‚Äî Defer rembg to a `BackgroundTask` on the legacy analyze path

The dominant cost on `/closet/analyze` was `_matte_crops` ‚Äî
`background_matting.matte_crop` runs **serially** (the comment in the
code is explicit: parallel rembg calls OOM in 3 GB pods). For a
2-garment fake outfit this added ~30 s of rembg time to a 17 s Eyes
window, total wall ~47 s.

The Phase-O.6 single-pass path already had the plumbing to defer
matting via a `_run_background_matte` FastAPI BackgroundTask. We
generalised it:

* `app/config.py` ‚Äî new flag `DEFER_REMBG_ON_ANALYZE` (default `True`).
* `app/services/garment_vision.py` ‚Äî `analyze_outfit` skips
  `_matte_crops` when the flag is on and tags every returned item
  with `defer_matte: True`.
* `app/api/v1/closet.py` ‚Äî `AnalyzeIn` whitelists the new
  `defer_matte` field in the response. `CreateItemIn` accepts
  `defer_matte: bool = False` from the frontend.
  `create_item` queues `_run_background_matte` when
  `from_one_pass OR defer_matte` is true.
* `frontend/src/pages/AddItem.jsx` ‚Äî captures `it.defer_matte`
  per item, propagates through `buildCreatePayload`.

End-to-end timing (verified live on a 2-garment fake outfit):

| Stage | Before | After |
| --- | ---: | ---: |
| `/analyze` wall | 47.7 s | **28.9 s** |
| Save ‚Üí first paint | instant | instant |
| Background matte completes | (in-line, already done) | +5‚Äì10 s after save |

The closet card thumbnail (raw bbox crop, Patch 7) renders
immediately; `bestImageUrl()` swaps it for the matted
`clean_image_url` as soon as the background task finishes.
**Kill-switch:** set `DEFER_REMBG_ON_ANALYZE=false` in the env to
restore the synchronous behaviour.

### Patch 9a ‚Äî Retire `EYES_ONE_PASS` from production

Three benchmark runs (`/app/scripts/run_eyes_benchmark.py` with
`--analyzer=one_pass`) on the CCP-Ninja 30-image set confirmed that
**Gemini-2.5-Flash returns exactly one garment per image** regardless
of prompt phrasing (three different prompt rewrites tested). On
multi-garment outfits this collapses recall to ~10 %.

* `app/api/v1/closet.py` ‚Äî removed the `if settings.EYES_ONE_PASS:`
  branch from the `/analyze` handler. The production path now
  unconditionally calls `analyze_outfit` (legacy SegFormer + per-crop
  Eyes), which scored mean IoU 0.71 / recall 0.41 on the same set ‚Äî
  the only architecture that produces multi-garment arrays in
  practice.
* `app/config.py` ‚Äî `EYES_ONE_PASS` setting deleted. Env var is no
  longer read; setting it in `.env` has no effect.
* `app/services/garment_vision.py` ‚Äî `analyze_outfit_one_pass`
  function retained for benchmark use; its docstring carries the
  retirement note + link to this session log.
* `deploy/.env.example` ‚Äî replaced the EXPERIMENTAL block with a
  RETIRED note.
* `closet.py:analyze` response ‚Äî `one_pass` field is now always
  `False` (kept for backwards-compat with older frontend bundles).

### Patch 10a ‚Äî Per-category min-area threshold in SegFormer parser

`clothing_parser.py:_MIN_AREA_FRAC = 0.005` filtered out any mask
covering less than 0.5 % of the image. In a full-body photo a
sunglasses / belt / fine purse mask is typically 0.05‚Äì0.3 % of the
frame ‚Üí vanished entirely ‚Üí CCP benchmark showed 0 % recall on
sunglasses, belt, bag, hat, purse despite SegFormer's classes
including all of them.

Refactored to per-category thresholds:

    _MIN_AREA_FRAC_DEFAULT = 0.005       # tops, bottoms, dresses
    _MIN_AREA_FRAC_PER_CATEGORY = {
        'accessory': 0.0005,  # sunglasses, belts, bags, scarves
        'footwear' : 0.0008,  # shoes / socks at full-body distance
        'headwear' : 0.0010,  # hats in wide shots
    }

Both call-sites in `clothing_parser.py` updated to call
`_min_area_frac_for(category)`.

Benchmark delta after Patch 10a (legacy analyzer, 15 images):

|                            | before | after |
| --- | ---: | ---: |
| mean IoU                   | 0.713  | **0.745** |
| recall @ 0.5               | 0.414  | **0.563** |
| precision @ 0.5            | 0.720  | **0.778** |
| sunglasses recall          | 0 %    | **80 %** |
| belt recall                | 0 %    | **50 %** |
| bag recall                 | 0 %    | **75 %** |
| hat recall                 | 0 %    | **100 %** |
| purse recall               | 0 %    | **50 %** |
